## 7. Offline Verifier

### 7.1 Inputs

The verifier takes (a) chain bytes over one or more `chain_scope` ranges, (b) a trust anchor describing acceptable signing public keys for principals appearing in the chain, (c) optionally, an **anchor trust anchor** for verifying ANCHOR blocks (Pass 4) â€” a *resolved* set of acceptable TSA signing certificates (public keys); in any-TSA deployments this is a pinned certificate set, and in eIDAS-qualified deployments it is resolved from the EU Trusted List (LOTL) **on the caller's side** and supplied already-resolved, so the verifier never fetches a trust list itself, (d) optionally, a **delegator-authority policy** â€” a mapping from the operator's capability identifiers to the action names a holder of that capability may delegate (consumed by Pass 5), (e) optionally, **bytes-binding disclosures** â€” for one or more documents, the bytes the verifying party holds plus the producer-side `nonce` disclosed in an evidence bundle (consumed by Pass 6), and (f) optionally, **QES validation verdicts** â€” for each ASSIGNMENT whose `delegator_authority.signature` carries a qualified (eIDAS) algorithm, a verdict produced by a conformant QES validator (reference: `@val-protocol/qes-validator`) run on the caller's side against the EU Trusted List; consumed by the Â§5.2 Profile-C delegation-trust-chain evaluation. Like the QTSP trust list, inputs (d), (e), and (f) are obtained and pinned independently of the chain bytes; without (d), Pass 5 still enforces carrier presence on v2 ASSIGNMENT bodies but cannot evaluate scope-versus-authority; without (e), Pass 6 reports `not_evaluated`; without (f), a qualified-algorithm block is classified `qualified_unverified` â€” the verifier never validates a qualified signature itself, exactly as it never fetches a trust list. None of these is an operator secret: a disclosure's `nonce` ships in the evidence bundle, not from a live call to the operator, and collision-resistance makes it a self-authenticating witness rather than a trust dependency.

### 7.2 Procedure

The verifier executes six passes; passes 1â€“3 are mandatory, passes 4, 5, and 6 are conditional.

**Pass 1 â€” Integrity.** Walk blocks in sequence order. For each block: recompute its `chain_hash` from the canonical preimage (Â§4.3); verify `previous_hash` matches the preceding block's `chain_hash`. Stop on first failure. Output: `integrity = green | red`, `first_break_at = <seq> | n/a`. *(VAL v0.1 defines no per-block signature in the wire format (Â§4.3); root human-binding strength is a conformance-profile property, Â§5.2.)*

**Pass 2 â€” Lineage.** For each block `B` of type not in `{ASSIGNMENT, ANCHOR}`: resolve `B.parent_assignment_hash`; assert the referenced ASSIGNMENT exists in the supplied chain bytes; recurse `parent_assignment_hash` up to root; assert root's principal is human per Â§5.2; assert depth â‰¤ 16. Output: `lineage = green | red`, `first_orphan_at = <seq> | n/a`.

**Pass 3 â€” Scope.** For each non-assignment block `B`: compute the effective scope of `B.parent_assignment_hash` (transitive intersection per Â§6.7); evaluate satisfaction per Â§6.6; if not satisfied, fail. Output: `scope = green | red`, `first_violation_at = <seq> | n/a`.

**Pass 4 â€” Anchor (optional).** For each ANCHOR block (Â§4.4): recompute the Merkle root over the in-band blocks `covered_range.from_sequence â€¦ to_sequence` using `merkle_alg` (`val.checkpoint-merkle.v1`, Â§8.1) and assert it equals `merkle_root`; parse the RFC 3161 TimeStampToken (`tst`) and assert `messageImprint.hashedMessage == merkle_root` (Â§8.1 â€” the root is the timestamped digest, not re-hashed); verify the token's CMS signature against the supplied anchor trust anchor (Â§7.1(c)) â€” the signature is over the DER-encoded `signedAttributes`, the `message-digest` signed attribute MUST equal `SHA-256(TSTInfo)`, and the signer certificate MUST carry the `id-kp-timeStamping` EKU; surface the attested `genTime`. Pass 4 asserts **temporal existence only** â€” that the covered range existed no later than `genTime`, bound to this chain â€” and evaluates no time *policy*. A chain MAY carry multiple ANCHOR blocks; each is verified against its own `covered_range`. Output: `anchorBinding = verified | mismatch | not_evaluated`. `verified` = at least one ANCHOR was present, the anchor trust anchor was supplied, and every ANCHOR's root and token verified; `mismatch` = an ANCHOR was present and supplied with a trust anchor but failed (root mismatch, broken `messageImprint` binding, or invalid signature) â€” sticky, a later valid ANCHOR does not clear it; `not_evaluated` = no ANCHOR block, or no anchor trust anchor supplied. A `not_evaluated` result never fails the verdict; anchoring is opt-in and additive.

**Pass 5 â€” Delegator authority.** For each ASSIGNMENT block (root or sub-assignment, whatever surface minted it): if the body version is â‰¥ 2, assert `human_attestation.delegator_authority` is present â€” absence fails the pass; if the body version is 1 and the carrier is absent, count the block as pre-carrier legacy without failing. For each block carrying the carrier, when the delegator-authority policy (Â§7.1(d)) is supplied: resolve `policy[delegator_authority.capability]` â€” an unknown capability fails the pass; assert `scope.act âŠ†` the resolved delegable-action set â€” any excess action fails the pass (authority escalation: the ASSIGNMENT delegates more than its issuer could grant). When the carrier's reserved `signature` sub-field is populated (Profiles B/C), the verifier additionally validates it against the profile's trust anchor; under Profile A the slot is absent and the authority claim carries the profile's operator-attested residual. Output: `authority = green | red | none`, `legacyPreAuthorityAssignmentCount = <uint>`. `none` means no ASSIGNMENT in the verified range engaged the pass.

**Well-known basis `container_owner` â€” re-derived where the chain permits.** `basis` values are operator-namespaced, but `container_owner` is RESERVED by this spec for the standing "the attesting human owns the from-scratch, participant-less container this ASSIGNMENT scopes" (e.g. a send/share container created by its sender). Because the claim is about the chain's own subjects, two checks are re-derivable from chain bytes alone, and a conforming verifier MUST apply them to every carrier with this basis â€” independent of whether a Â§7.1(d) policy is supplied: (a) `delegator_authority.scope_ref` MUST equal the ASSIGNMENT's `scope.res.in_workspace` â€” the claimed authority is scoped to the very container the ASSIGNMENT scopes; (b) for each COMMUNICATION block whose `parent_assignment_hash` is this ASSIGNMENT and whose `principal` is a human (`user:<id>`), `SHA-256(<id>)` MUST equal the ASSIGNMENT's `human_attestation.subject_user_hash` â€” the act was performed by the attested owner, not relabelled. Either failure fails the pass. An `agent:` principal has no second chain occurrence of the delegating human to cross-check; its ownership claim carries the Profile-A operator-attested residual like any other basis, until the Profile B/C `signature` binding ships. Implemented in the reference verifier as of `@val-protocol/chain-verifier` 0.3.0.

**Well-known basis `ceremony_session_delegated` â€” reserved (2026-07-01).** RESERVED by this spec for the standing "the delegating human's entitlement over the scoped container was attested by an authorized operator principal via a single-use ceremony session" â€” the account-less signer completing a hosted ceremony being the motivating case. Two checks are re-derivable from chain bytes, and a conforming verifier MUST apply them to every carrier with this basis: (a) `delegator_authority.scope_ref` MUST equal the ASSIGNMENT's `scope.res.in_workspace` (same re-derivation shape as `container_owner`); (b) the carrier MUST co-occur with a populated **qualified** `delegator_authority.signature` â€” an account-less authority claim without a qualified instrument fails the pass. The carrier SHOULD name its attestor (`attested_by`, an operator-namespaced subject hash of the session creator) and the session (`session_ref`); these are tamper-evident, chain-embedded attestations â€” NOT offline-provable entitlement. The entitlement claim carries the attestor's operator-attested residual: the provable ceiling is *"a qualified natural person signed this exact delegation,"* never *"they were entitled to."* *(Reference status: verifier support pending; a verifier that does not yet implement these checks treats the basis as an ordinary operator-namespaced label per the general rule.)*

**Pass 6 â€” Bytes-binding (optional).** For each MUTATION carrying `bytes_commitment` (Â§4.4): if a bytes-binding disclosure (Â§7.1(e)) is supplied for the block's `resource.resource_id`, compute `c' = SHA-256(disclosed_bytes)`, then `C' = SHA-256("val.bytes-commitment.v1" â€– 0x00 â€– disclosed_nonce â€– c')`, and assert `C'` equals the block's `bytes_commitment.value`. The verifier hashes the disclosed bytes itself â€” it MUST NOT accept a caller-supplied hash in place of the bytes. Output: `bytesBinding = bound | mismatch | not_evaluated`. `bound` = at least one commitment was disclosed and every disclosed one reproduced; `mismatch` = a disclosed commitment failed to reproduce (the bytes in hand are not the committed document) â€” sticky, a later match does not clear it; `not_evaluated` = no MUTATION carried both a commitment and a matching disclosure. This pass is the **bytes-binding rail**: it answers "the content-address on the chain *is this document in hand*," a property distinct from grounding ("the actor read content-address X before deriving from it", Â§7.5) â€” the two are orthogonal. A commitment with no disclosure, or a disclosure for an uncommitted block, is `not_evaluated` and **never fails** the verdict; bytes-binding is opt-in and additive. Because the on-chain `value` is a *hiding* commitment (the nonce is producer-side, never on chain or in any export), an exported chain reveals nothing about the bytes and is not a cross-tenant confirmation oracle. Implemented in the reference verifier as of `@val-protocol/chain-verifier` 0.7.0.

### 7.3 Output

A verification report â€” the `ValVerificationResult` shape the reference verifier (`@val-protocol/chain-verifier`) returns, as a JSON object:

```
{
  "integrity":              "green" | "red",
  "lineage":                "green" | "red",
  "scope":                  "green" | "red",
  "grounding":              "green" | "red",
  "authority":              "green" | "red" | "none",
  "anchorBinding":          "verified" | "mismatch" | "not_evaluated",
  "conformanceProfile":     "A" | "B" | "C" | "unknown",
  "profilesPresent":        [ "A" | "B" | "C", ... ],
  "firstLineageViolation":  { "sequenceNumber": <tstr>, "reason": <tstr> } | null,

  "firstGroundingViolation":{ "sequenceNumber": <tstr>, "reason": <tstr> } | null,
  "firstAuthorityViolation":{ "sequenceNumber": <tstr>, "reason": <tstr> } | null,
  "firstAnchorViolation":   { "sequenceNumber": <tstr>, "reason": <tstr> } | null,
  "anchors":                [ { "sequenceNumber": <tstr>, "genTime": <tstr>, "covered_range": { "from_sequence": <tstr>, "to_sequence": <tstr> } } ],

  "nonValBlockCount":       <uint>

```

`anchorBinding` (Â§7.2 Pass 4) reports external-anchor verification; the `anchors` array surfaces, per verified ANCHOR, the TSA-attested `genTime` and the covered range (empty when Pass 4 is `not_evaluated`). These fields are additive: a verifier predating Pass 4, or a chain with no ANCHOR block, reports `not_evaluated` with an empty `anchors` array and the core verdict is unaffected.

`conformanceProfile` is the **floor** â€” the weakest profile among root ASSIGNMENTs in the verified range (Â§5.2); `profilesPresent` itemizes every profile observed. A verifier MUST NOT round the chain-level profile up: reporting the strongest profile present would let one qualified grant mask a chain of operator-attested ones, the exact over-claim the `rootSubject.source` verbatim rule exists to prevent. *(Reference status: published `@val-protocol/chain-verifier` â‰¤ 0.9.x predates this amendment â€” it reports the strongest profile present and does not emit `profilesPresent`; until the floor release ships, consumers MUST read `rootSubject.source` and per-assignment outcomes rather than the chain-level letter.)* A deployment that wraps the report for transport MAY add envelope fields (verifier identity, timestamp, a signature over the report); those are not part of the reference output.

All four properties green (integrity, lineage, scope, grounding) â€” plus `authority` green or `none` (Pass 5) â€” is the protocol's affirmative statement. The semantic content of "lineage green" depends on the conformance profile per Â§5.2: under Profile A, it means every action traces to a human-attributed authorization within the tamper-evident chain; under Profiles B and C, it means every action traces to a cryptographically-signed authorization by an identified natural person. Verifier consumers MUST read the `conformanceProfile` field alongside the lineage result to interpret the residual-trust statement correctly.

### 7.4 Reference Implementation

`@val-protocol/chain-verifier` (Apache-2.0) is the protocol's reference implementation. Passes 1 (integrity), 2 (lineage), and 3 (scope â€” including the Â§6.4 Merkle isolation-membership check) ship as `verifyChain` + `verifyValChain`, alongside a **grounding** re-derivation (Â§7.5): a MUTATION that cites `grounded_document_hashes` must cite content the same principal already read (via a prior ACCESS) in the chain â€” re-derived from the chain bytes alone, independent of any substrate enforcement.

The reference verifier distinguishes **conformance profiles A, B, and C** (Â§5.2). Profile A (operator-attested, chained) is bound by the *presence* of `human_attestation` at the root; **Profile B (WebAuthn device key) is fully verified** â€” the delegator signature is cryptographically checked (`verifyDelegatorSignature`) and chained to the enrolled, self-attested org-root key (`verifyDelegationTrustChain`); **Profile C (qualified / eIDAS) is verified when a QES validation verdict is supplied** (Â§7.1(f)) â€” the delegation trust chain returns `authority_verified_qualified`; absent a verdict it is classified by algorithm and returns `qualified_unverified`, never silently upgraded. Full Profile-C verification is the two-package story: `@val-protocol/chain-verifier` + `@val-protocol/qes-validator` + EU Trusted List access. **CONSENT and COMMUNICATION carry dedicated verifier passes** (a per-action signature check and a container-owner re-derivation respectively) and are verified if present, though the reference producer does not emit them. **Pass 4 (anchor) ships in `verifyValChain` as of `@val-protocol/chain-verifier` 0.8.0** â€” it parses an ANCHOR block's RFC 3161 TimeStampToken, recomputes the `val.checkpoint-merkle.v1` root over the covered range, checks the `messageImprint` binding, and verifies the token's CMS signature against a supplied anchor trust anchor (Â§7.1(c)), reporting `anchorBinding`. **The SETTLEMENT block type is not yet implemented** â€” SETTLEMENT is recognized only for generic lineage/scope. A conforming implementation MAY emit any reserved type ahead of this reference. The block types shipped end-to-end (reference producer + verifier) today are ASSIGNMENT, ACCESS, and MUTATION; ANCHOR is verifier-implemented and operator-emitted.

**Pass 6 (bytes-binding)** ships in `verifyValChain` as of `@val-protocol/chain-verifier` 0.7.0 via the optional `bytesDisclosures` input (Â§7.1(e)), reporting `bytesBinding` (`bound` / `mismatch` / `not_evaluated`). It is opt-in and additive: it never affects the four core properties, and a verifier predating it ignores the `bytes_commitment` field per the Â§4.4 unknown-field rule.
